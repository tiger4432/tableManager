# [Task] Cursor-based Pagination 아키텍처 도입 (데이터 누락/중복 완벽 방어)

## 1. 🚨 현재 상황 (AS-IS) 및 한계
현재 클라이언트와 서버는 데이터를 주고받을 때 **Offset-based Pagination (`skip`, `limit`)** 방식을 사용하고 있습니다.
* **증상 (데이터 뛰어넘기/누락)**: 다중 사용자 환경에서 내가 스크롤을 내려 다음 데이터를 요청하는 찰나의 순간에, 내 위에 있던 데이터가 누군가에 의해 '삭제'되면 서버 쪽 전체 인덱스가 위로 당겨집니다. 결과적으로 나는 당겨진 인덱스만큼의 데이터를 영영 볼 수 없게 되는 구멍(Gap)이 발생합니다.
* **증상 (데이터 중복)**: 반대로 내 위에 데이터가 '추가'되면 인덱스가 밀려서, 아까 본 데이터를 한 번 더 응답받게 됩니다. (이 부분은 현재 클라이언트의 `row_id` 중복 검사 로직으로 간신히 방어 중입니다.)
* **웹소켓의 한계**: 실시간 삭제/추가 이벤트가 브로드캐스트되어 로컬 인덱스를 보정해주고 있지만, "API 요청 전송 후 ~ 응답 수신 전" 사이의 0.01초 레이스 컨디션은 방어할 수 없습니다.

## 2. 🎯 달성 목표 (TO-BE)
**"서버의 중간 데이터가 수만 개가 지워지든 생기든 상관없이, 절대로 이빨이 빠지지 않는 완벽한 무한 스크롤 구현"**

## 3. 💡 해결 아이디어: 커서 기반 페이징 (Cursor-based Pagination)

* **핵심 개념**: 절대적인 숫자인 `skip` 대신, 내가 마지막으로 눈으로 확인한 데이터의 **고유 ID(`cursor`)** 를 기준점으로 삼습니다.
* **통신 스펙 변경안**:
    * **AS-IS**: `GET /data?skip=100&limit=50` (앞에서부터 100개 건너뛰고 50개 줘)
    * **TO-BE**: `GET /data?cursor_row_id=uuid-1234&limit=50` (내가 마지막으로 본 데이터가 uuid-1234야. 걔 **다음** 것부터 50개 줘)
* **서버 처리 방식 (`server/main.py`)**:
    1. 데이터베이스에서 `uuid-1234` 행의 정렬 기준 값(예: `updated_at`, `row_id`)을 조회.
    2. 그 정렬 값보다 **작거나/큰** 행들만 필터링해서 `limit`만큼 반환.
    3. 이렇게 하면 중간에 누군가 지워져도, 기준점(`uuid-1234`) 자체는 변하지 않으므로 그 다음 데이터를 정확히 이어갈 수 있습니다.

## 4. 📝 세부 구현 방법론 (How-to)

### [1] API 통신 스펙 변경
* 클라이언트 `fetchMore` 요청 시, 단순히 `skip=100`을 보내는 대신 현재 화면의 맨 마지막 데이터(혹은 맨 처음 데이터)의 `row_id`를 추출해 `cursor` 파라미터로 보냅니다.
* 예시: `/data?cursor=uuid-1234&limit=50&direction=forward`

### [2] 서버 쿼리 최적화 (`server/main.py`)
현재 서버의 점프 로직은 `COUNT(*)`로 위치를 찾은 뒤 `OFFSET`을 쓰는 "무늬만 커서" 방식입니다. 이를 진짜 커서 방식으로 바꿔야 합니다.
* **AS-IS (점프 로직의 한계)**: `row_id`로 몇 번째 행인지 센다(`COUNT`) -> 그 숫자만큼 건너뛴다(`OFFSET`). (매번 전체 테이블 스캔 발생)
* **TO-BE (진짜 커서 쿼리)**: 
  ```python
  # 1. 커서 행의 기준값 획득
  cursor_row = db.query(DataRow).filter_by(row_id=req.cursor).first()
  cursor_val = cursor_row.updated_at
  
  # 2. OFFSET 없이 인덱스를 직접 타는 고속 쿼리 실행
  if req.order_desc:
      query = query.filter(or_(
          DataRow.updated_at < cursor_val,
          and_(DataRow.updated_at == cursor_val, DataRow.row_id < req.cursor)
      ))
  ```

### [3] 점프(Jump) 기능과의 분리
* 점프 기능은 임의의 행으로 건너뛰어야 하므로 어쩔 수 없이 해당 행의 절대 위치를 알아내기 위한 `COUNT(*)` 연산이 1회 필요합니다. (사용 빈도가 낮아 성능상 허용됨)
* 하지만 일반 스크롤링은 마우스를 내릴 때마다 수십 번 호출되므로 절대 `COUNT(*)`나 `OFFSET`을 사용하면 안 되며, 위 [2]번 항목의 쿼리를 사용하도록 파이프라인을 분리해야 합니다.

## 5. 📝 진행 시 고려사항 (Impact)
* 이 변경은 클라이언트(`table_model.py`)의 `fetchMore` 요청 구조와 백엔드(`main.py`)의 쿼리 로직을 동시에 뒤엎는 **메이저 아키텍처 개편**입니다.
* 정렬 조건(`order_by`, `order_desc`)이 달라질 때마다 커서 대소 비교(>, <) 방향이 달라지므로 서버 쿼리 빌딩 로직이 꽤 복잡해집니다.
