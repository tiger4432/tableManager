# [Task] Total Count 실시간 진성 동기화 아키텍처 고민

## 1. 🚨 현재 상황 및 한계
현재 테이블의 총 데이터 개수(`_total_count`)는 서버가 DB 부하를 줄이기 위해 5초간 캐싱(`TABLE_COUNT_CACHE`)을 수행하고 있습니다.
* **증상**: 대규모 삭제/생성 직후에 클라이언트가 데이터를 페치하면 옛날 개수(캐시)를 받게 되어 무한 페치 루프나 UI 어긋남이 발생합니다.
* **임시 조치**: 클라이언트 단에서 `req_skip`과 `new` 데이터 존재 여부를 비교해, 데이터가 비어 있으면 강제로 모델 크기를 줄여버리는 긴급 방어선(Emergency Brake)을 구축해 두었습니다.

## 2. 🎯 달성 목표
**"클라이언트 방어 코드에 의존하지 않고, 항상 서버가 100% 정확한 total_count를 제공할 수 있는 방법 모색"**

## 3. 💡 해결 아이디어 (브레인스토밍)

### [아이디어 A] 웹소켓을 통한 증분(Delta) 브로드캐스팅
* 서버가 행을 삭제하거나 추가할 때마다 DB에서 `count(*)`를 다시 세지 말고, 애플리케이션 메모리 상에서 캐시된 `total` 값을 산술적으로 `+1` 또는 `-N` 연산한 뒤, 클라이언트들에게 `event: total_count_update` 형태로 쏴줍니다.
* **장점**: DB 부하 0, 클라이언트 즉각 반응
* **단점**: 다중 인스턴스 환경이거나, 다른 필터 조건(`search_query`)이 걸려있는 클라이언트에게는 이 방식이 적용될 수 없음. (필터에 맞는 증감량은 `count(*)` 없이 알 수 없기 때문)

### [아이디어 B] 비동기 백그라운드 카운터 도입
* 클라이언트가 `_refresh_total_count`를 요청하면 무조건 5초 캐시된 값을 주되, 캐시가 만료되었거나 강제 갱신 트리거(웹소켓 이벤트)가 발생하면 백그라운드 태스크(Celery/BackgroundTasks)에서 `count(*)`를 실행합니다.
* 실행 완료 후 웹소켓을 통해 해당 테이블을 구독 중인 클라이언트들에게 정확한 새 `total` 값을 푸시합니다.

### [아이디어 C] DB 트리거 및 집계 테이블 활용 (Materialized View)
* 테이블의 전체 개수나 주요 부서별 개수를 저장해두는 별도의 메타데이터 테이블(또는 Redis)을 구축합니다.
* 데이터가 삽입/삭제될 때마다 DB 트리거 혹은 애플리케이션 레벨에서 해당 집계 테이블을 O(1) 성능으로 업데이트합니다.
* 클라이언트는 항상 이 집계 테이블에서 `total_count`를 가져가므로, 캐싱 딜레이 없이 100% 정확한 값을 즉시 제공받습니다.

---

## 4. 📝 향후 진행 방향
이 태스크를 진행하게 될 때, 현재 서비스의 테이블 데이터 규모(백만 건 이상 여부)와 필터링 기능의 사용 빈도를 고려하여 가장 적합한 아키텍처를 선정하고 구현합니다.
