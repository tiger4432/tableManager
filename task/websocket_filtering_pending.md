# [Backlog] 필터 상태에서의 실시간 웹소켓 동기화 개선 플랜

## 1. 🚨 문제 현상
현재 `table_model.py`의 `_on_websocket_broadcast` 로직은 수신된 이벤트를 무조건 현재 화면(`_data`)에 밀어 넣고 있습니다.
만약 사용자가 **"사과"를 검색(필터링) 중인데 누군가 "바나나" 데이터를 생성**했다면, 필터 조건과 무관하게 사용자의 화면 맨 위나 맨 아래에 "바나나"가 삽입되어 버리는 심각한 UX/데이터 정합성 오류가 발생할 수 있습니다.

## 2. 🔍 원인 및 제약 사항
* 클라이언트(PyQt)는 사용자가 입력한 검색어(`self._search_query`)가 서버의 복잡한 SQL LIKE 조건(다중 컬럼 검색)에 부합하는지 스스로 판단할 능력이 없습니다.
* 따라서 새로 생성되거나 수정된 데이터가 현재 필터 조건에 맞는지 틀린 지는 **서버만** 알고 있습니다.

## 3. 🛠️ 해결 방안 (제안)

**원칙: 검색/필터링 중일 때는 클라이언트가 함부로 데이터를 삽입/삭제하지 않고, 기존 데이터의 "값 갱신"만 허용하며 나머지는 서버에 재검증을 요청한다.**

### [1] `batch_row_create` 방어
* **상황**: 누군가 새로운 행을 생성함.
* **대책**: 현재 `self._search_query`가 존재한다면, `self.beginInsertRows`를 통한 강제 삽입을 **생략(Skip)** 합니다. 대신 `self._refresh_total_count()`만 호출합니다. 만약 생성된 데이터가 검색 조건에 부합한다면 서버가 `total_count`를 늘려줄 것이고, 이때 클라이언트는 남은 공간을 채우기 위해 정식으로 `Fetch` 파이프라인을 가동하여 데이터를 안전하게 가져오게 됩니다.

### [2] `batch_row_upsert` (수정 및 삽입) 방어
* **상황**: 누군가 데이터를 수정하거나 엑셀에서 대량 덮어쓰기함.
* **대책**: 
    1. 수신된 데이터가 이미 화면에 보이는 데이터(`idx is not None`)라면: 단순히 값만 업데이트하고 주황색으로 하이라이팅합니다. (검색 조건에서 벗어나더라도 당장 화면에서 없애면 UX 혼란이 올 수 있으므로 값만 갱신)
    2. 수신된 데이터가 화면에 없는 새로운 데이터(`idx is None`)라면: `self._search_query`가 존재할 경우 맨 위에 강제 삽입하는 행위를 **차단**합니다.

### [3] `batch_row_delete` 방어
* **상황**: 누군가 데이터를 삭제함.
* **대책**: 삭제는 검색 조건과 무관하게 화면에 존재하는 데이터라면 지워버리는 것이 맞으므로 기존 로직(화면에 있으면 삭제)을 그대로 유지합니다.
