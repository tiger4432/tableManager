# [Version / Fix] 2026.04.19 (Late) - 실시간 데이터 업데이트 시 커서 위치 고정 및 정밀 추적 확보

## 📝 개요
클라이언트 테이블 상에서 사용자가 특정 셀의 데이터를 직접 변경했을 때 완료 직후 발생하는 '커서 및 스크롤 초기화(튕김) 현상'을 전면 수정했습니다. 방대한 양의 데이터를 다룰 때 편집 맥락을 유지시켜주는 매우 결정적인 UI/UX 패치입니다.

## 🛠️ 주요 변경 사항

### 1. `client/models/table_model.py`: 파괴적인 `beginResetModel()` 제거 및 정밀 `beginMoveRows()` 도입
*   **배경**: WebSocket을 통해 실시간 데이터 병합(batch_row_upsert) 명령이 전달될 때, 행 자체의 순서가 뒤얽힐 가능성이 있다는 이유로 Qt의 전체 화면 지우개 역할을 하는 `beginResetModel()` 함수가 사용되고 있었습니다. 이 함수가 호출되면 QTableView는 현재 저장된 스크롤의 위치와 유저가 클릭해두었던 커서 선택 영역을 모두 파괴하고 무조건 화면 최상단 0번째 뷰포트로 강제 귀환하는 고질적 결함을 야기했습니다.
*   **해결**:
    *   `beginResetModel()`을 전면 삭제하고, 정밀 타격 모델인 **`beginMoveRows()`** 및 데이터 개별 갱신 시그널인 **`self.dataChanged.emit()`**로 핵심 엔진을 교체했습니다.
    *   최신순 부상 설정(Floating)이 꺼져있을 때(`sort_latest == False`), **화면 이동 및 커서 초기화가 일체 발생하지 않으며**, 수정한 셀의 색상만 변경되고 유저는 500번째, 1000번째의 스크롤 위치에 그대로 안착해 작업을 이어갈 수 있습니다.

### 2. `client/main.py`: `rowsMoved` 시그널 연계를 통한 셀 추적 레이더 적용
*   **배경**: 최신순 부상 설정(Floating)이 켜져 있을 경우(`sort_latest == True`), 유저가 셀을 수정하면 그 행은 전체 테이블의 0번 인덱스(맨 윗줄)로 로켓처럼 솟아오릅니다. 이전 모델에서는 억지로 커서를 위로 돌렸기 때문에 내가 무슨 데이터로 수정했는지 직관적으로 추적할 수 없었습니다.
*   **해결**: 
    *   Qt 내부 이벤트인 `rowsMoved` 시그널을 강제로 인터셉트하는 `_on_rows_moved` 이벤트를 `main.py`에 새롭게 매핑했습니다.
    *   이제 행이 0번 혹은 어느 곳으로 이동(Move)하든 간에 뷰포트(`scrollTo`)가 그 이동된 데이터 셀의 인덱스 좌표값을 정확하게 포착하고 렌더링 화면을 스크롤하여 **도착지점(행)의 위치까지 카메라(Cursor)를 강제로 밀착 이동**시킵니다. 수정한 데이터를 절대 시야에서 놓칠 일이 사라졌습니다.
