# 프로젝트 변경 이력 (2026.04.20) - Phase 73.5

## 📝 개요
고속 검색 및 정렬 시 발생하는 데이터 정합성 이슈를 해결하고, 히스토리 패널의 노이즈를 제거하여 시스템 안정성을 확보했습니다.

---

## 🚀 주요 변경 사항

### 1. 검색 세션 가드 (Search Session Guard) 도입
- **문제**: 비동기 검색 요청 중 사용자가 검색어를 빠르게 바꿀 경우, 이전 검색어의 결과가 나중에 도착하여 현재 테이블 데이터를 덮어씌우는 '고스트 데이터' 현상 발생.
- **해결**: `uuid`를 활용한 세션 관리 시스템 구축.
  - `ApiLazyTableModel`에 `_search_session_id` 필드 추가.
  - 검색(`set_search_query`) 및 정렬(`set_sort_latest`) 변경 시마다 새 UUID 생성.
  - 비동기 워커(`ApiFetchWorker`, `ApiTargetedRowIdWorker`)가 세션 ID를 파라미터로 들고 갔다가 결과와 함께 반환.
  - 응답 수신 시 세션 ID가 현재 모델의 ID와 일치하지 않으면 즉시 폐기.

### 2. 히스토리 패널 노이즈 차단 및 요약 최적화
- **문제**: 배경 데이터 페칭(Lazy Loading)이나 원격 업데이트 브로드캐스트 처리 시 발생하는 내부 `dataChanged` 시그널이 히스토리에 로컬 수정으로 기록되어 사용자 혼란 가중.
- **해결**: 
  - `_handle_data_changed`에 `_fetching`, `_is_processing_remote`, `_first_fetch` 상태에 따른 엄격한 차단 가드(Guard) 적용.
  - 대량 삭제(`batch_row_delete`) 이벤트 시 개별 행 로깅 대신 "n개의 행 삭제됨" 형태의 요약 로그로 전환하여 UI 부하 감소 및 가독성 확보.

### 3. UI 카운터 동기화 정합성 확보
- **문제**: 필터링 조건 변경 시 프록시 모델의 `rowCount()` 계산이 완료되기 전에 UI 레이블이 먼저 갱신되어 잠깐 숫자가 틀리게 보이는 현상.
- **해결**: 
  - `QTimer.singleShot(10, ...)`을 활용해 10ms의 미세 지연 후 카운트를 갱신하도록 설계.
  - 필터 툴바뿐만 아니라 메인 윈도우의 상태바 카운터도 동일한 로직으로 동기화.

## ✅ 검증 결과
- **고속 검색 테스트**: "321" 입력 후 빠르게 지울 시, 이전 "321" 검색 결과가 뒤늦게 나타나지 않음 확인.
- **히스토리 상쾌함**: 하단 스크롤로 데이터 확장 시 히스토리에 아무런 노이즈가 남지 않음 확인.
- **카운터 정확도**: 필터링 시 숫자가 튀지 않고 정확한 매칭 결과 표시 확인.
