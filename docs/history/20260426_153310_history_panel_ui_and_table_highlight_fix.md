# 히스토리 패널 UI 개선 및 테이블 셀 변경 시 전체 로우 하이라이팅 버그 수정

## 개요
- **일자**: 2026년 04월 26일
- **주요 내용**: 히스토리 리스트 패널의 시각적 밀집도 및 가독성 최적화, 그리고 클라이언트 측 WebSocket 처리 로직 결함으로 인해 데이터 변경 시 로우 전체가 하이라이팅되는 문제 수정.

## 주요 변경 사항

### 1. 히스토리 패널 UI/UX 개선 (`panel_history.py`, `history_logic.py`)
- **다중 건 트랜잭션 텍스트 단일 라인 통일**: 확장형/요약형 트랜잭션 로우의 텍스트에서 불필요한 줄바꿈(`\n`)을 제거하여 한 줄(Single-Line)로 일관되게 렌더링되도록 수정했습니다.
- **아이템 간격(Spacing) 제거 및 균일화**: `QListView`의 `setSpacing(0)` 및 `setUniformItemSizes(True)` 설정을 통해 로우 간의 간격을 없애고, 행 높이를 통일시켜 시각적 밀집도를 높였습니다. 여백은 CSS `padding`을 통해 제어하도록 개선했습니다.

### 2. 테이블 전체 하이라이팅 버그 수정 (`table_model.py`)
- **현상**: 사용자가 특정 셀 하나를 수정할 때마다 WebSocket 실시간 동기화 브로드캐스트가 수신되면서 해당 로우(행)의 전체 셀이 모두 오렌지색(`is_overwrite = True`)으로 변하는 치명적인 시각적 오류가 발생했습니다.
- **원인**: 클라이언트 `table_model.py` 내의 `_on_websocket_broadcast` 로직에서 `batch_row_upsert` 이벤트를 처리할 때, 서버가 보내준 전체 최신 행 상태(`new_data`)에 루프를 돌면서 모든 컬럼에 하드코딩된 `is_overwrite = True` 속성을 강제 주입하는 잘못된 로직이 존재했습니다.
- **해결**: 클라이언트의 강제 하이라이팅 루프를 제거하고, 서버(`crud.py`)에서 판별하여 전송한 정밀한 속성(수정된 컬럼만 `is_overwrite = True`)을 클라이언트가 그대로 신뢰하고 교체(`row["data"] = new_data`)하도록 아키텍처를 정상화했습니다.

### 3. 히스토리 트랜잭션 그룹핑 최적화 (`HistoryDataManager` 등)
- **현상 및 배경**: 기존에는 대량의 데이터(수만 건 이상의 행)가 실시간으로 유입되거나 일괄 수정될 때, 히스토리 리스트에 개별 셀 업데이트 로그가 낱개로 생성되어 성능 저하 및 UI 프리징(Hang)을 유발했습니다.
- **해결 로직**: 
  - 백엔드와 프론트엔드의 비동기 파이프라인에서 **Transaction ID 기반의 그룹핑 기능**을 완벽하게 통합했습니다.
  - 대량 업로드나 동시 업데이트 발생 시, 동일한 트랜잭션으로 묶인 로그들은 단일 부모 항목(Summary Item)으로 압축 표출되도록 하여 렌더링 리소스를 최적화했습니다.
  - 사용자가 요약 항목을 더블클릭(`toggle_expand`) 시, `is_fetching` 락(Lock)을 통해 중복 요청을 방지하며 하위 상세 내역을 레이지 로딩(Lazy Loading)으로 가져와 펼쳐보는 구조로 안정화했습니다.
  - 서버 응답 누락으로 인한 무한 대기를 방지하기 위한 타임아웃 메커니즘과, Garbage Collection에 의한 비동기 콜백 유실 방지를 위해 콜백을 Bound Method로 강제하는 정책을 적용했습니다.

## 기대 효과
- **히스토리 패널 사용성 증대**: 컴팩트하고 일관적인 높이의 리스트 뷰를 통해, 대량의 이력도 스크롤 공간 낭비 없이 직관적으로 파악할 수 있습니다.
- **정확한 시각적 피드백 제공**: 의도치 않은 전체 로우 오염을 방지하고, 정확히 사용자가 수정한 셀 하나만 시각적으로 강조되도록 본래의 의도를 완벽히 복원했습니다.
