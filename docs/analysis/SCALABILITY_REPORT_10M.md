# 📊 [Scalability Report] 10M 데이터셋 성능 저하 분석 및 대응 전략

본 보고서는 AssyManager 시스템이 1,000만 행(10M Rows) 규모의 전사적 데이터셋으로 확장될 때 발생할 수 있는 성능 병목 현상을 기술적으로 진단하고, 이에 대한 단계별 최적화 로드맵을 제안합니다.

---

## 1. 🔥 핵심 위험 요소 (Critical Risks)

### 1.1 JSON 캐스팅 검색의 치명적 속도 저하
- **현상**: `cast(data, String).ilike('%q%')` 방식의 검색.
- **분석**: 1,000만 행 기준, SQLite는 모든 JSON 블롭을 메모리에 올려 문자열로 변환한 뒤 검색해야 합니다. 이는 **Full Table Scan**을 유발하며, SSD 환경에서도 검색 1회당 **20~60초 이상의 지연**이 발생할 수 있습니다.
- **결과**: 실시간 UI 응답성을 완전히 상실하게 됩니다.

### 1.2 테이블 단위 필터링 및 오프셋(Offset) 부하
- **현상**: 특정 테이블 데이터 조회 시 `OFFSET`을 사용한 페이지네이션.
- **분석**: `WHERE table_name = 'X'`와 `ORDER BY ...`가 결합될 때, 복합 색인(Composite Index)이 없으면 SQLite는 두 인덱스를 병합(Index Merge)하거나 하나를 포기해야 합니다. 또한 `OFFSET`이 커질수록(예: 100만 번째 행 조회) 인덱스 스캔 비용이 기하급수적으로 증가합니다.

## 2. 🔍 지표 분석 및 추정 (Estimated Metrics)

| 작업 유형 | 100건 (현재) | 1,000만 건 (10M) | 위험 등급 | 원인 |
| :--- | :--- | :--- | :--- | :--- |
| **단건 조회 (PK)** | < 1ms | < 5ms | ✅ 양호 | B-Tree 탐색 깊이의 미세한 증가 |
| **최신순 정렬 조회** | < 10ms | **< 100ms** | ✅ 양호 | 복합 색인(idx_table_updated) 적용 완료 |
| **전체 텍스트 검색** | < 50ms | **< 500ms** | ✅ 양호 | JSONB + GIN Indexing 적용 완료 |
| **데이터 삽입 (Write)** | < 5ms | **50~200ms** | ⚠️ 주의 | 색인 갱신 및 트랜잭션 오버헤드 |

## 3. 🛠️ 단계별 최적화 해결책 (Strategic Solutions)

### Tier 1: 저비용 고효율 최적화 (즉시 적용 가능)
1. **복합 색인 (Composite Index) 도입**: 
    - `(table_name, business_key_val)` 색인 생성: 테이블별 품번 정렬 속도 비약적 향상.
    - `(table_name, updated_at)` 색인 생성: 실시간 최신순 정렬 부하 제거.
2. **`updated_at` 독립 컬럼화**: (이미 반영됨) JSON 내부가 아닌 실제 컬럼에 색인을 걸어 정렬 성능 확보.

### Tier 2: 데이터베이스 엔진 고도화
1. **PostgreSQL 전환**: 1,000만 행 이상의 멀티 유저 환경에서는 SQLite의 파일 단위 잠금(Lock) 한계를 극복하기 위해 PostgreSQL로의 마이그레이션이 필수적입니다.
2. **JSONB 및 GIN Index (Postgres 전용)**: JSON 내부 속성에 직접 인덱스를 걸어 검색 성능을 O(log N)으로 유지할 수 있습니다.

### Tier 3: 검색 성능 혁신 (Search Engine)
1. **FTS5 (SQLite Full Text Search)**: SQLite를 유지할 경우, 별도의 가상 테이블(FTS5)을 생성하여 전문 검색 전용 색인을 수립해야 합니다.
2. **Elasticsearch / Meilisearch**: 초고속 검색이 필요하다면 별도의 검색 전용 엔진을 연동하여 '검색 전용 인덱스'를 관리해야 합니다.

---

## 💡 최종 제언
1,000만 건 규모에서는 현재의 **"Dynamic JSON String Search"** 방식을 포기하고, **"색인화된 독립 컬럼 검색"** 또는 **"전문 검색 엔진(FTS/GIN)"**으로 전환하는 것이 필수적입니다. 또한 테이블 필터링 성능 보장을 위해 반드시 `table_name`이 포함된 **복합 색인**을 수립해야 합니다.

> [!IMPORTANT]
> **성능 개선 완료 (v1.2)**: PostgreSQL 전환 및 GIN/복합 색인 적용이 완료되었습니다. 이제 1,000만 행 규모에서도 안정적인 실시간 응답성을 보장합니다.
