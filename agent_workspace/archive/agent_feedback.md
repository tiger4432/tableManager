# Sub-Agent 병렬 개발 피드백 및 보완 가이드라인 

PM(리드 에이전트)이 A, B 에이전트의 구현 문서를 검토한 결과, 서로 맞물리는 부분에서 심각한 앱 뻗음(UI Freeze) 및 과부하가 예상됩니다. 아래 피드백 구문을 각 에이전트에게 프롬프트로 전달하여 코드를 수정하게 하십시오.

---

## 📢 [To. 에이전트 A]: 셀 동기화 및 비동기 처리 보완
**현재 문제점**: 작성한 내용 중 `server/database/crud.py` 구현체가 워크스페이스에 없고, 클라이언트 `setData` 내의 네트워크 요청(`urllib.request`)이 **동기식(Sync)**으로 처리되어 응답이 오기 전까지 데스크톱 창 전체가 멈추는(Freeze) 현상이 발생할 수 있습니다.

> **[A 에이전트용 수정 프롬프트]**
> "PM의 검토 피드백이야. 다음 세 가지를 보완해줘:
> 1. `server/database/crud.py` 파일을 생성하고 `update_cell` 로직을 작성해줘. JSON 컬럼 업데이트 시 SQLAlchemy의 `flag_modified`를 반드시 사용해야 해.
> 2. `client/models/table_model.py`의 `setData` 내 네트워크 요청 로직을 동기식 `urllib.request`가 아닌 **Python `threading` 혹은 PyQt의 백그라운드 Worker 스레드 기반**의 비동기 방식으로 리팩토링해서 UI 프리징을 막아줘.
> 3. 에이전트 B가 수십 개의 셀을 한 번에 붙여넣을 수 있도록 **다중 셀 일괄 업데이트(Batch Update)용 REST API** (`PUT /tables/{table_name}/cells/batch`)를 서버 쪽에 추가 신설해줘."

---

## 📢 [To. 에이전트 B]: 다중 복사/붙여넣기 네트워크 스팸 방지
**현재 문제점**: 엑셀에서 수백 건의 셀을 복사하여 붙여넣을 때, TSV를 파싱한 뒤 반복문 안에서 `setData`를 개별적으로 연속 호출하게 됩니다. 이 방식은 에이전트 A의 백그라운드 서버 업데이트 요청을 수백 번 연달아 쏘게 만들어 Ddos 공격처럼 서버와 클라이언트 모두를 마비시킵니다.

> **[B 에이전트용 수정 프롬프트]**
> "PM의 검토 피드백이야. 다중 붙여넣기 기능 시 서버 과부하를 막기 위해 구조를 변경해줘:
> 1. `keyPressEvent`에서 Ctrl+V로 생성된 수십 개의 데이터를 파싱한 뒤, 곧바로 `setData` 루프를 돌리지 마.
> 2. 대신 한 번에 묶어서(JSON Array 객체 형태) A 에이전트가 만들어 둘 **`Batch Update API`**로 단 1번만 네트워크 전송을 수행해줘.
> 3. 서버 응답이 성공(Success)으로 돌아오면, 그때 모델의 `self._data`를 메모리 상에서만 일괄 덮어쓰고 최상단/최하단 index에 대해 `dataChanged` 시그널을 딱 한 번만 방출하게 해서 렌더링 최적화를 해줘."
