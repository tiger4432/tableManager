# Agent D v3 작업 보고서 - UI 가시성 및 WebSocket 최적화

## 1. 수행 작업 요약
- **WebSocket 공유화**: 탭마다 개별 생성되던 `WsListenerThread`를 `MainWindow` 수준의 단일 리스너로 통합하여 점유 자원을 최적화했습니다.
- **UI 기동 이슈 해결**: 서버에서 테이블 목록 수신 후 탭을 생성하고 첫 탭을 포커싱하는 로직을 정교화하여 빈 화면이 노출되는 버그를 수정했습니다.
- **스키마 동적 로딩 개선**: 비동기 워커(QRunnable)의 수명 주기를 `MainWindow`에서 관리하도록 변경하여, 기동 직후 컬럼 정보가 디폴트(`id`, `name`, `status`)로 고착되던 현상을 해결했습니다.
- **안정성 강화**: 불완전하게 동작하던 "(빈 테이블)" 표시 로직을 제거하고, 레이아웃 및 시그널 연결을 복구하여 전반적인 UI 안정성을 높였습니다.

## 2. 기술적 변경 사항
### Client (`main.py`)
- `_active_models`: 현재 열려 있는 모든 `ApiLazyTableModel` 인스턴스를 관리합니다.
- `_start_shared_ws()`: 앱 기동 시 단 하나의 WebSocket 연결만 수립합니다.
- `_dispatch_ws_message()`: 수신된 JSON 데이터를 활성 모델들에게 분배합니다.
- `_load_table_schema()`: 각 모델별 컬럼 정보를 안전하게 순차/병렬 로드합니다.

### Client Model (`table_model.py`)
- 개별 WS 리스너 로직 제거 및 외부 디스패처 수용 구조로 변경.
- 컬럼 업데이트 시 `print` 로그 추가로 로딩 상태 가시화.

## 3. 검증 결과
- **WS 연결 단일화**: 서버 로그를 통해 여러 탭을 열어도 단일 소켓만 유지됨을 확인.
- **초기 로딩**: 서버 내 모든 테이블이 탭으로 정상 생성되며, 각기 다른 컬럼 스키마가 로드됨.
- **실시간 동기화**: 타 클라이언트에서의 변경 사항이 중앙 리스너를 거쳐 해당 탭에 즉시 반영됨.

## 4. 향후 계획
- 대규모 테이블(수만 행 이상)에서의 메모리 사용률 모니터링 및 추가 최적화 고려.
- 사용자 정의 컬럼 순서 저장 기능 검토.
